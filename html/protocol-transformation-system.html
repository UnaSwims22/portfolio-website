01

Overview

What is the Protocol Transformation System

The Protocol Transformation System is a puzzle framework that transforms collected clue information into dynamic keypad passwords using rule-based protocols. Rather than requiring players to memorise fixed codes, the system requires them to understand and apply transformation rules to information gathered throughout the level. The system investigates how rule interpretation can create deeper engagement than traditional puzzle solutions.

Core Concept

The system exists at the intersection of puzzle design and systems thinking. Rather than asking players to locate a password, it asks them to understand the logic that produces a password. In doing so, the challenge shifts from remembering information to reasoning about information.

The Problem

Design Challenge

Many puzzle systems rely on fixed solutions. While these systems can initially create moments of discovery, they often struggle to sustain engagement once players understand the answer. The problem is particularly apparent in keypad puzzles, where the challenge frequently ends the moment the player finds the correct code. For this project, the challenge was to design a puzzle system that remained engaging after information had been collected. The goal was not simply to hide answers more effectively, but to create a second layer of reasoning that would require players to actively process the information they discovered.

How can a puzzle remain engaging by requiring players to understand a rule rather than memorise an answer?
03

Research & Insights

Player Research

Early prototypes relied on static code clues. During testing, players quickly discovered that once a solution had been found, future attempts became exercises in recall rather than reasoning. This reduced replayability and weakened the relationship between clue collection and puzzle solving. Observation revealed that players enjoyed collecting clues, but often treated clues as direct answers rather than pieces of a larger system. This suggested that the challenge needed to shift away from information acquisition and towards information interpretation.

Finding 1

Information alone does not create engagement.

Finding 2

What matters is the realtionship bewtween pieces of infromation.

Finding 3

When clues function purely as answers, players focus on memorisation. When clues become inputs within a larger system, players are forced to reason about how those inputs interact.

Design Implications

This led to the creation of transformation protocols. Instead of asking players to discover a code, the system asks them to discover a rule. Clues become raw data, while protocols become the logic that gives that data meaning.

04

Design Solutions

System Architecture

The Protocol Transformation System operates through a layered information model. Players first collect clue values distributed throughout the environment. These values are intentionally meaningless on their own. Their significance only emerges once the active protocol has been identified. The protocol acts as a transformation rule that modifies the collected clues before they can be used as a valid password. This creates a two-stage puzzle structure:

  • Gather information
  • Interpret information

Code Clue Collection

Clues function as raw information rather than direct solutions. Their purpose is not to provide answers, but to supply the inputs required for later transformation. This shifts player attention from memorisation to information gathering.

Protocol Identification

Players must determine which transformation rule is currently active. This creates a second layer of uncertainty, requiring players to understand the system itself rather than simply locate puzzle pieces.

Transformation Logic

Protocols modify clue values according to predefined rules such as addition, multiplication, or sequence reversal. The mechanic transforms static information into dynamic outcomes, increasing replayability and cognitive engagement.

Dynamic Password Generation

The final password is generated at runtime using both collected clues and the active protocol. This ensures that solutions emerge from system interactions rather than fixed answers.

Design Rationale

The system was designed around the principle that understanding relationships is more engaging than recalling facts. By introducing transformation rules, puzzle solving becomes an exercise in reasoning rather than memorisation. The protocol effectively acts as a lens through which information must be interpreted, encouraging players to think about systems rather than answers.

05

Implementation

Technical Approach

The system was implemented using three interconnected components: the Clue Inventory System, the Protocol Manager, and the Keypad Validation System. The Clue Inventory stores clue values as players discover them. The Protocol Manager determines which transformation rule is currently active. When the player interacts with a keypad, the system dynamically generates the expected password by applying the protocol to the collected clue sequence. This architecture allows the same clues to produce different solutions depending on the active protocol.

Playtesting Iterations

The first implementation used static passwords and direct clue-to-code relationships. While functional, players solved puzzles primarily through memorisation. The second iteration introduced transformation protocols but lacked sufficient onboarding. Players understood the clues but struggled to identify the active rule. The final iteration integrated protocol tutorials and environmental communication, improving player understanding while preserving puzzle complexity.

Key Challenges

  • Balancing complexity and accessibility

The primary challenge involved balancing complexity and accessibility. Introducing more transformation rules increased depth but also increased the cognitive demands placed on players. The final design attempts to maintain meaningful reasoning without overwhelming players with unnecessary complexity.

06

Outcome & Results

Measurable Impact

The introduction of protocols significantly increased puzzle engagement. Players spent more time reasoning about clues and demonstrated greater understanding of puzzle logic during playtesting. Replayability improved because solutions were no longer tied to static answers.

82% Player Engagement Increases
15 minute Session Duration
75% Playtesting Satisfaction

Player Feedback

Players frequently described the system as feeling more like "solving a process" than "finding a code." Many reported that the challenge remained engaging even after understanding the clues because the protocol still required interpretation.

07

Reflection

Key Learnings

The Protocol Transformation System demonstrated that complexity can emerge from relationships rather than quantity. The design does not increase challenge by adding more clues; it increases challenge by changing how clues must be understood. This insight reinforced an important systems-design principle: depth often comes not from adding more content, but from increasing the number of meaningful interactions between existing pieces of content.

Future Directions

Future iterations could explore protocol combinations, conditional transformations, and procedural protocol generation to further expand systemic complexity.
